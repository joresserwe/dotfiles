# Claude Code 작성 가이드

> 참고: <https://github.com/ykdojo/claude-code-tips>

---

## 1. Claude Code는 챗봇이 아닙니다

많은 사람이 Claude Code를 "질문하면 답하는 챗봇"처럼 씁니다. 실제 동작 방식은 완전히 다릅니다.

Claude Code는 하나의 루프를 반복합니다:

```
컨텍스트 수집 → 행동 → 결과 검증 → 완료 or 다시 수집
```

프롬프트를 던지면 Claude가 바로 코드를 짜는 게 아닙니다. 먼저 필요한 파일을 읽고(수집), 코드를 수정하고(행동), 수정이 맞는지 테스트를 돌리고(검증), 문제가 있으면 다시 파일을 읽으러 돌아갑니다(다시 수집).

이 루프를 **에이전트 루프**라고 부릅니다. 핵심은 이거예요:

> Claude Code는 한 번에 답을 주는 게 아니라, 여러 번 돌면서 스스로 고쳐나갑니다.

루프 중간중간에 사람이 개입할 수 있습니다. "잠깐, 그 방향 아니야"라고 끼어들거나, 추가 정보를 줄 수 있습니다.

## 2. 문제가 생기면 어디를 봐야 하나: 5개 Surface

결과가 불안정할 때 대부분 프롬프트를 고치지만, Tw93은 **5개 레이어 중 어디가 문제인지 먼저 찾으라**고 합니다.

### ① Context Surface (맥락 층)

> "Claude가 지금 뭘 알고 뭘 모르는 상태인가?"

CLAUDE.md, rules, 메모리, Skills 설명 등이 너무 많으면 정작 중요한 정보가 묻힙니다.

### ② Action Surface (행동 층)

> "Claude가 지금 뭘 할 수 있는 상태인가?"

기본 내장 도구 + MCP 외부 도구 + 플러그인. 도구를 너무 많이 붙이면 도구 정의 자체가 컨텍스트를 잡아먹어서 성능이 떨어집니다.

### ③ Control Surface (제어 층)

> "Claude가 하면 안 되는 행동을 어떻게 막고 있는가?"

permissions, sandbox, hooks로 제어합니다.

### ④ Isolation Surface (격리 층)

> "이 작업을 메인 세션과 분리해서 돌려야 하나?"

Subagent, Worktree, Forked Session이 여기 속합니다.

### ⑤ Verification Surface (검증 층)

> "작업이 끝났는지, 결과를 믿을 수 있는지 어떻게 아는가?"

tests, lint, 스크린샷, 로그, CI 등으로 확인합니다.

### 요약

| 증상 | 확인할 Surface |
|---|---|
| 결과가 불안정하면 | Context |
| 엉뚱한 행동을 하면 | Control |
| 자동화가 폭주하면 | Isolation |
| "됐다"고 했는데 안 됐으면 | Verification |

## 3. 개념 경계: 비슷해 보이는데 다 다릅니다

| 개념 | 역할 | 주의사항 |
|---|---|---|
| **CLAUDE.md** | 프로젝트의 "헌법". 세션 시작 시 제일 먼저 읽는 파일 | 팀 위키처럼 모든 지식을 넣지 말 것. 짧고 핵심만 |
| **.claude/rules/** | 상황별 세부 규칙 (특정 폴더/언어에만 적용) | 모든 규칙을 CLAUDE.md에 몰아넣지 말고 폴더별로 분리 |
| **Built-in Tools** | 파일 읽기/쓰기, 터미널 실행, 검색 등 기본 내장 도구 | 외부 연동은 MCP로 분리 |
| **MCP** | GitHub, Sentry, DB 같은 외부 서비스 연결 통로 | 서버 5개만 붙여도 컨텍스트의 12.5%가 도구 정의에 소모 |
| **Skill** | "이런 상황에서는 이렇게 해라" — 일하는 절차 매뉴얼 | 백과사전이 아닌 "절차"를 담는 것 |
| **Hook** | 무조건 실행되는 강제 규칙 (Claude가 피할 수 없음) | 모든 판단을 Hook으로 대체하지 말 것. 세밀한 판단은 Skill로 |
| **Subagent** | 격리된 컨텍스트에서 독립적으로 동작하는 미니 Claude | 5개 이상 동시에 돌리면 상태 관리가 어려움 |
| **Plugin** | Skills + Hooks + MCP를 묶어서 배포하는 패키지 | 런타임 기본 기능이 아닌 배포용 |

**한 문장 정리:**

- 새 동작 능력 → Tool / MCP
- 일하는 방법 → Skill
- 강제 제약 → Hook
- 격리 실행 → Subagent
- 프로젝트 간 배포 → Plugin

## 4. 컨텍스트: 가장 많은 사람이 가장 크게 착각하는 부분

> "200K 토큰이면 충분하잖아?"

아닙니다. 200K 중 실제로 쓸 수 있는 건 **160~180K** 정도. 나머지는 Claude Code를 돌리기 위한 고정 비용입니다.

### 고정 오버헤드 (15~20K 토큰)

코드 한 줄 읽기 전에 이미 차 있는 영역.

| 항목 | 토큰 | 설명 |
|---|---|---|
| 시스템 지시 | ~2K | Claude Code 자체의 기본 작동 규칙 |
| Skill 설명 | 1~5K | 활성화된 모든 Skill의 "언제 쓸지" 설명 |
| MCP 서버 도구 정의 | 10~20K | 숨은 킬러. MCP 서버 1개당 ~5K |
| LSP 상태 | 2~5K | IDE 연동 시 코드 에러 정보 등 |

> MCP 서버 5개 연결하면? 고정 오버헤드만 25,000 토큰. 전체의 12.5%가 코드 한 줄 읽기 전에 사라집니다.

### 반고정 (5~10K 토큰)

프로젝트마다 다르지만 세션 내내 유지됨.

| 항목 | 토큰 | 설명 |
|---|---|---|
| CLAUDE.md | 2~5K | 프로젝트 규칙 |
| Memory | 1~2K | Claude가 기억하는 과거 맥락 |

### 동적 가용 (160~180K 토큰)

실제로 일하는 공간. 대화가 길어지면 점점 줄어듭니다.

| 항목 | 설명 |
|---|---|
| 대화 히스토리 | 지금까지 주고받은 대화 |
| 파일 내용 | Claude가 읽은 코드 |
| 도구 호출 결과 | grep, bash 등의 출력 |

### 컨텍스트 분층 원칙

| 분류 | 대상 | 설명 |
|---|---|---|
| **항상 상주** | CLAUDE.md, MCP 도구 정의, 시스템 지시 | 세션 시작부터 끝까지. 길게 쓰면 안 됨 |
| **경로별 로드** | `.claude/rules/` | 특정 폴더에서 작업할 때만 로드 |
| **필요 시 로드** | Skills | 설명만 상주, 필요 시 전체 내용 로드 |
| **격리 로드** | Subagents | 메인 컨텍스트와 완전 분리. 독립된 200K |
| **컨텍스트 바깥** | Hooks | 토큰 0 소비. 외부에서 셸 스크립트처럼 실행 |

Skill 설명은 짧게:

- 나쁜 예: *"이 스킬은 Rust 프로젝트의 코드 변경을 리뷰합니다..."* (~45 토큰)
- 좋은 예: *"PR 리뷰 시 정확성 중심으로 사용."* (~9 토큰)

## 5. 압축(Compaction)의 함정

대화가 길어지면 Claude Code가 자동으로 오래된 대화를 요약해서 공간을 만듭니다. 문제는 **"다시 읽을 수 있는 것"부터 삭제**한다는 것. 도구 출력(Tool Output)이 먼저 날아갑니다.

### 해결책 1: Compact Instructions

CLAUDE.md에 명시:

```markdown
## Compact Instructions
- 아키텍처 결정 사항은 절대 요약에서 생략하지 말 것
- 수정된 파일 목록은 유지할 것
- 테스트 통과/실패 상태는 보존할 것
```

### 해결책 2: HANDOFF.md

긴 세션을 끝내기 전에 Claude에게 작업 내용을 정리시킵니다:

> "지금까지 한 작업을 HANDOFF.md에 정리해. 뭘 했고, 뭘 시도했고, 뭐가 됐고 안 됐고, 다음에 뭘 해야 하는지."

다음 세션에서 HANDOFF.md만 읽고 바로 이어서 작업할 수 있습니다.

## 6. 도구 설계: 에이전트용 도구는 다르게 만들어야 합니다

사람용 API와 에이전트용 도구는 설계가 달라야 합니다. AI 에이전트는 "이걸 쓸까 말까"를 매번 판단해야 하기 때문입니다.

### AskUserQuestion 시행착오

| 시도 | 방식 | 결과 |
|---|---|---|
| 1차 | 기존 도구에 question 파라미터 추가 | 실패 — Claude가 무시하고 계속 진행 |
| 2차 | 특정 마크다운 포맷으로 출력하라고 규칙 부여 | 반쯤 실패 — 포맷을 자주 까먹음 |
| 3차 | 완전히 독립된 도구로 분리, 호출 시 즉시 멈춤 | 성공 |

**교훈:** 에이전트에게는 "쉽게 맞게 쓸 수 있는" 도구를 만들어야 합니다.

### 검색 도구

- RAG(벡터 검색) → Claude가 결과를 잘 활용 못함
- Grep 도구로 바꿔서 Claude가 직접 검색어를 만들게 함 → 효과 급상승
- 부수적 발견: Claude가 Skill 파일 → 참조 파일 → 또 참조 파일을 따라가는 패턴 → **점진적 공개(progressive disclosure)**

## 7. Hooks: "매번 기억하길 기대하지 마세요"

CLAUDE.md에 "파일 저장 후 prettier를 돌려라"라고 적어도 Claude가 가끔 까먹습니다. Hook은 이런 걸 **확정적으로** 처리합니다.

### Hook 이벤트

| 이벤트 | 설명 |
|---|---|
| `PreToolUse` | 도구 실행 전에 끼어들기 (차단 가능) |
| `PostToolUse` | 도구 실행 후 자동 실행 |
| `PostToolUseFailure` | 도구 실패 시 실행 |
| `Notification` | 알림 보낼 때 |
| `UserPromptSubmit` | 사용자가 프롬프트를 보낼 때 |

### 세 겹 방어

| 조합 | 문제 |
|---|---|
| CLAUDE.md만 | Claude가 자주 못 본 척 |
| Hook만 | 세밀한 판단이 안 됨 |
| Skill만 | 강제력이 없음 |
| **셋이 같이** | 비로소 안정적 |

> Hook 출력은 `head -30`으로 제한하세요. 너무 길면 컨텍스트를 오염시킵니다.

## 8. 프롬프트 캐싱: 비용을 결정하는 숨은 핵심

### 프롬프트가 쌓이는 순서

1. **Base System Instructions** → 전역 캐시 (바뀌지 않으면 재사용)
2. **Tools 정의** → 전역 캐시
3. **CLAUDE.md & Memory** → 프로젝트별 캐시
4. **Session State** → 세션별 캐시
5. **Messages (대화 히스토리)** → 매 턴마다 증가

1~4는 바뀌지 않으면 캐시되어 비용이 **1/10로** 절감. 하나라도 바뀌면 그 지점부터 캐시가 전부 깨집니다.

### 캐시를 깨뜨리는 실수들

- 시스템 프롬프트에 타임스탬프 넣기
- 도구 정의 순서를 비결정적으로 섞기
- 세션 중간에 MCP 서버 추가/삭제
- 세션 중간에 모델 변경 (Opus → Haiku 전환 시 캐시를 처음부터 새로 생성)

> 현재 시간 같은 동적 정보는 시스템 프롬프트가 아닌 **사용자 메시지**에 넣을 것.

### 도구 지연 로딩 (defer_loading)

MCP 도구가 수십 개일 때 이름만 있는 경량 스텁을 먼저 보내고, Claude가 "이 도구 쓸래"라고 하면 그때 전체 스키마를 로드. 캐시 프리픽스는 안정적으로 유지됩니다.

## 9. CLAUDE.md 잘 쓰는 법

### 넣어야 할 것

- 빌드/테스트/실행 방법 (`npm run build`, `pytest` 등)
- 핵심 디렉토리 구조와 모듈 경계
- 코드 스타일과 명명 규칙
- 환경 관련 함정 ("M1 Mac에서 이 라이브러리는 arm64로 빌드해야 함")
- NEVER 리스트 (절대 하면 안 되는 것들)
- Compact Instructions (압축 시 지켜야 할 것들)

### 넣으면 안 되는 것

- 긴 배경 설명 ("이 프로젝트는 2019년에 시작되어...")
- 전체 API 문서 (Claude가 필요하면 직접 읽게 하세요)
- "좋은 코드를 작성하라" 같은 공허한 원칙
- Claude가 레포를 읽으면 알 수 있는 뻔한 정보
- 저빈도 작업 지식 (Skills로 분리)

> Anthropic 자체 CLAUDE.md가 약 **2.5K 토큰**. 이걸 기준으로 삼으면 됩니다.

**팁:** Claude가 실수할 때마다 *"다시는 이 실수 안 하도록 CLAUDE.md를 업데이트해"*라고 시키면 시간이 갈수록 같은 실수가 줄어듭니다. 다만 주기적으로 리뷰해서 오래된 제약이 아직 유효한지 확인하세요.

## 10. 잘 안 알려진 유용한 명령어들

| 명령어 | 설명 |
|---|---|
| `/simplify` | 방금 수정한 코드를 재사용/품질/효율 세 축으로 검사하고 바로 고침 |
| `/rewind` | 특정 체크포인트로 돌아가서 다시 시작. 잘못된 길로 너무 깊이 갔을 때 |
| `/btw` | 메인 작업 중단 없이 사이드 질문 |
| `/insight` | 현재 세션을 분석해서 CLAUDE.md에 넣을 만한 것들을 추출 |
| `ESC` x 2 | 이전 입력으로 돌아가서 수정 후 재전송 |

> 대화 기록은 `~/.claude/projects/` 아래에 `.jsonl`로 저장되어 있어서 grep으로 검색 가능합니다.

## 11. 세 단계의 성장

| 단계 | 질문 |
|---|---|
| 1단계: 기능 탐색 | "이거 어떻게 쓰지?" |
| 2단계: 시스템 구축 | "어떻게 조합하지?" |
| 3단계: 거버넌스 | "제약 안에서 어떻게 알아서 돌아가게 하지?" |

대부분 1단계에 머물면서 "더 좋은 프롬프트"를 찾습니다. 진짜 생산성이 뛰는 건 **2~3단계**로 넘어갈 때입니다.

### Health Check

```bash
npx skills add tw93/claude-health
```

`/health`를 치면 자기 Claude Code 설정이 어디가 부실한지 리포트가 나옵니다.
